# 실제 단위 표시 설계 — 추상 수치를 친숙한 물리 단위로

- 작성일: 2026-07-17
- 상태: 승인됨 (구현 대기)
- 범위: UI 개선 — 시뮬레이션의 추상 단위 수치(질량·길이·속력·시간)를 실제 물리 단위(태양질량·목성질량·지구질량·kg, km·AU, 광속의 %, 초·년)로 표시. **표시만 바꾸며 엔진·물리는 건드리지 않는다.**
- 선행: 블랙홀 4덩어리(이벤트→효과·디테일 팩·중력 렌즈·조석 파괴) 완료.

> 이 문서는 살아있는 설계 문서다. 구현 중 설계가 바뀌면 코드와 함께 이 문서를 갱신한다.

## 1. 목적과 원칙

지금 UI는 추상 단위(질량 20.00, 반지름 7.82 등)를 그대로 보여줘 감이 오지 않는다. 이를 친숙한 실제 단위로 바꿔 직관을 준다.

이 시뮬레이션은 **일부러 비물리적인 상수**를 쓴다(광속 `C=25`로 낮춰 블랙홀을 손에 잡히는 크기로 만듦). 따라서 완벽하게 자기무결한 실제 단위계는 불가능하다. 대신 **몇 개의 앵커를 잡아** 나머지를 파생시키고, 수치가 직관적으로 "그럴듯하게" 느껴지게 한다. 절대적 정확성보다 감이 오는 스케일이 목표다.

**앵커 (사람이 승인함):**
- **질량:** 항성 프리셋(`BODY_PRESETS.star.mass = 2000`) = **1 태양질량**. → 붕괴 임계(3000)가 1.5 태양질량이 되어, 실제 별의 블랙홀 붕괴 한계(찬드라세카르 ~1.4배)와 근사 일치.
- **길이:** 항성 반지름(`radiusFromMass(2000) ≈ 7.815`) = **1 태양반지름**. → 밀도 일정 가정이 "모든 천체가 태양 밀도"인 일관된 우주가 된다.
- **속력:** `C = 25` = **광속**. → 속력을 "광속의 %"로 표시. 이 우주는 C가 작아 공전 속도가 상대론적이라 %c가 가장 정직하다.
- **시간:** 위에서 파생(길이 ÷ 속력). 경과·증발 시간을 실제 초로 환산.

**받아들인 한계 (정직하게 표기):**
- 시뮬레이션 질량비가 압축돼 있어(소행성 0.5 : 항성 2000), 항성=태양이면 **가장 작은 천체도 지구질량급**이다(최소 스폰 질량 0.1 ≈ 지구질량 16배). 그래서 질량은 실질적으로 지구질량 이상 단위를 주로 쓴다. kg·톤 단위는 완성도를 위해 사다리에 두되 거의 쓰이지 않는다.
- C가 작은 탓에 1태양질량 블랙홀의 사건의 지평선이 ~57만 km로 나온다(실제 3km). 이는 "이 우주의 블랙홀은 원래 거대하다"는 설계를 정직하게 반영한다.

## 2. 앵커 상수와 스케일 (`lib/sim/realunits.ts`)

**실제 상수 (SI):**
```
SOLAR_MASS_KG    = 1.989e30
JUPITER_MASS_KG  = 1.898e27
EARTH_MASS_KG    = 5.972e24
SOLAR_RADIUS_KM  = 6.96e5
AU_KM            = 1.496e8
LIGHT_SPEED_KMS  = 2.998e5
```

**시뮬레이션 앵커에서 파생한 스케일:**
```
STAR_MASS        = BODY_PRESETS.star.mass                 // 2000, 앵커 참조(DRY)
MASS_SCALE_KG    = SOLAR_MASS_KG / STAR_MASS              // ≈ 9.945e26 kg / 시뮬 질량 1
STAR_RADIUS_SIM  = radiusFromMass(STAR_MASS)             // ≈ 7.815
LENGTH_SCALE_KM  = SOLAR_RADIUS_KM / STAR_RADIUS_SIM      // ≈ 89,060 km / 시뮬 길이 1
SPEED_SCALE_KMS  = LIGHT_SPEED_KMS / C                    // ≈ 11,992 km/s / 시뮬 속력 1
TIME_SCALE_S     = LENGTH_SCALE_KM / SPEED_SCALE_KMS      // ≈ 7.43 s / 시뮬 시간 1
```

`STAR_MASS`·`STAR_RADIUS_SIM`은 `BODY_PRESETS`/`radiusFromMass`를 참조해 프리셋이 바뀌면 앵커가 따라 움직인다. `C`는 `units.ts`의 기존 상수를 쓴다.

## 3. 적응형 포매터 (순수 함수)

각 함수는 시뮬 값을 받아, 크기에 맞는 단위를 골라 사람이 읽기 좋은 범위(대략 0.1~1000)로 문자열을 반환한다.

### `formatMass(simMass: number): string`
`kg = simMass · MASS_SCALE_KG` 계산 후, 값이 1 이상이 되는 가장 큰 단위를 고른다(내림차순 탐색):
```
태양질량(SOLAR_MASS_KG) → 목성질량(JUPITER_MASS_KG) → 지구질량(EARTH_MASS_KG) → 톤(1e3 kg) → kg
```
예: 항성 2000 → "1.0 태양질량", 행성 20 → "10.5 목성질량", 소행성 0.5 → "83 지구질량", 최소 0.1 → "17 지구질량". 값은 2~3 유효숫자(예: ≥100은 정수, ≥10은 소수 1자리, 그 미만은 2자리).

### `formatLength(simLength: number): string`
`km = simLength · LENGTH_SCALE_KM`. `km ≥ 1e6`이면 AU(`km / AU_KM`), 아니면 km. km는 천 단위 구분(예: "570,000 km"), AU는 소수 2~3자리(예: "0.12 AU").
예: 사건의 지평선(6.4) → "570,000 km", 항성 반지름(7.82) → "696,000 km"(=1 태양반지름), 행성 공전 거리(200) → "0.12 AU".

### `formatSpeed(simSpeed: number): string`
`frac = simSpeed / C`. `frac ≥ 0.01`(광속 1% 이상)이면 "광속의 {frac·100}%"(유효숫자 2자리), 미만이면 km/s(`simSpeed · SPEED_SCALE_KMS`, 정수 + 천 단위 구분).
예: 공전 행성(√10≈3.16) → "광속의 13%", 느린 천체(0.05) → "600 km/s".

### `formatTime(simSeconds: number): string`
`s = simSeconds · TIME_SCALE_S`를 초→분→시간→일→년으로 접는다. 년이 1e6 초과면 "사실상 영원". (기존 `BodyCard`의 증발 시간 접기 로직을 이 함수로 옮겨 재사용.)

## 4. 적용 범위 (표시만 교체, 로직 불변)

- **`components/ui/BodyCard.tsx`**
  - 질량 → `formatMass`, 반지름 → `formatLength`, 속력 → `formatSpeed`.
  - 블랙홀: 사건의 지평선·흡수 반경(ISCO) → `formatLength`. 증발까지 → `formatTime`(기존 `formatEvaporation`를 `formatTime` 재사용으로 대체). 시간 지연(×)은 무차원이라 그대로.
  - 질량 슬라이더 라벨 → `formatMass`. (슬라이더 값 자체는 추상 질량 그대로 조작.)
- **`components/ui/StatsHud.tsx`** — 경과 시간(`stats.simTime`) → `formatTime`.
- **`components/ui/SpawnPanel.tsx`** — 질량 슬라이더 라벨 → `formatMass`. 프리셋 버튼 아래 실제 질량 부제(예: "행성 · 10.5 목성질량").

원본 추상 숫자는 표시하지 않는다.

## 5. 파일과 테스트

**생성:**
- `lib/sim/realunits.ts` — 앵커 상수·스케일·4개 포매터.
- `lib/sim/realunits.test.ts` — Vitest.

**수정:**
- `components/ui/BodyCard.tsx`, `components/ui/StatsHud.tsx`, `components/ui/SpawnPanel.tsx` — 포매터 사용.

**테스트(`lib/sim/realunits.test.ts`):**
- 앵커 검증: `formatMass(2000)`이 "1.0 태양질량", 붕괴 임계 `formatMass(3000)`이 ~"1.5 태양질량", `formatLength(radiusFromMass(2000))`이 ~"696,000 km"(1 태양반지름), `formatSpeed(C)`가 "광속의 100%".
- 사다리 경계: 질량이 태양↔목성↔지구 단위로 전환되는 값, 길이가 km↔AU로 전환되는 값(`1e6 km`), 속력이 %c↔km/s로 전환되는 값(`frac=0.01`).
- 시간 접기: 초/분/시간/일/년 각 구간, "사실상 영원" 상한.
- UI(카드·통계·스폰)는 사람이 브라우저로 확인 — 씬 자체는 안 바뀐다.

## 6. 비목표

- **엔진·물리 변경 없음.** 내부 계산은 추상 단위 그대로. 변환은 표시 순간에만. 결정론·시뮬레이션 동작 불변.
- **자기무결한 실제 물리 단위계 아님.** C가 비물리적이라 블랙홀 길이 등 일부는 실제와 크게 다르다(의도된 한계, §1).
- **밀도 다양성 없음.** 밀도 일정 가정 유지(모든 천체 태양 밀도). 천체별 실제 밀도는 도입하지 않는다.
- **단위 토글/설정 없음.** 적응형 자동 선택 하나로 간다(YAGNI). 사용자가 kg↔태양질량을 수동 전환하는 UI는 만들지 않는다.

## 7. 조정 가능한 숫자와 한계

- 앵커는 프리셋(항성=태양)·`C`(=광속)에서 파생 — 프리셋을 바꾸면 스케일이 따라간다.
- 사다리 전환 임계(길이 `1e6 km`, 속력 `frac 0.01`)와 유효숫자 자릿수 — 가독성 조정 대상.
- §1의 두 한계(작은 천체가 지구질량급, 블랙홀 길이가 큼)는 시뮬레이션 설계의 산물이라 이 작업에서 고치지 않는다.

## 8. 사람의 브라우저 확인 (자동화 불가)

- 천체를 선택하면 카드의 질량·반지름·속력이 친숙한 단위로 뜨는가(예: 항성 "1.0 태양질량", 반지름 "696,000 km").
- 블랙홀 카드의 사건의 지평선·ISCO가 km/AU로, 증발까지가 실제 시간으로 뜨는가.
- 공전 중인 천체의 속력이 "광속의 X%"로, 느린 천체는 km/s로 뜨는가.
- 상단 경과 시간이 실제 시간 단위로 뜨는가.
- 스폰 패널의 질량 슬라이더 라벨·프리셋 부제가 실제 질량으로 뜨는가.
- 숫자가 한눈에 읽히는 범위(너무 길거나 0.000… 아님)인가.
