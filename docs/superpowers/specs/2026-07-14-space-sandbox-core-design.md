# 우주 샌드박스 — 1단계 코어 설계

- 작성일: 2026-07-14
- 상태: 승인됨 (구현 대기)
- 범위: 4단계 로드맵 중 **1단계(코어 샌드박스)만**

> 이 문서는 살아있는 설계 문서다. 구현 중 설계가 바뀌면 코드와 함께 이 문서를 갱신한다.

## 1. 목적

천체를 던지고 그것들이 서로 끌어당기다 뭉치는 과정을 구경하는 **샌드박스 장난감**. 교육용 정확성이나 실제 태양계 재현이 목적이 아니다. 물리는 "그럴듯하고 안정적"이면 충분하며, 재미와 반응성이 우선한다.

## 2. 로드맵과 이번 범위

4개 레이어를 순차적으로 쌓는다. 각 단계는 그 자체로 완성되어 즐길 수 있고, 다음 단계는 앞 단계를 부수지 않고 얹힌다.

| 단계 | 내용 | 상태 |
|---|---|---|
| 1 | **코어 샌드박스** — N-body 중력, 던지기, 병합, 카메라, 시간 제어 | 이번 스펙 |
| 2 | 신의 손 — 선택한 천체의 질량 편집·삭제·블랙홀화 | 이후 |
| 3 | 시나리오 프리셋 — 안정된 태양계/쌍성계 로드 후 망가뜨리기 (세이브/로드) | 이후 |
| 4 | 우주선 조종 — 플레이어 선박, 추력, 스윙바이 | 이후 |

순서의 근거: 2·3·4는 모두 1의 물리 상태 위에서만 성립한다. 2는 코어의 상태 조작 API를 그대로 쓴다. 3의 세이브/로드는 상태 직렬화이므로 2에서 조작 API가 정리된 뒤라야 깔끔하다. 4는 카메라 모드와 입력 체계가 새로 붙는 가장 이질적인 조각이라 마지막이다.

**규모:** 동시 수백 개(상한 512). 근사 없는 진짜 N-body가 가능한 규모이며, 개별 천체를 클릭해 만질 수 있는 상한이기도 하다.

## 3. 아키텍처

```
┌─ DOM 오버레이 (React + Tailwind) ──── UI 상태만 소유
├─ Canvas 경계 (React Three Fiber) ──── 엔진을 읽어서 그리기만
└─ 시뮬레이션 엔진 (순수 TypeScript) ── 모든 물리 상태 소유
```

물리를 메인 스레드의 프레임워크 무관 클래스로 둔다. Web Worker나 GPGPU는 이 규모에서 얻는 게 없고(YAGNI), 특히 GPGPU는 2단계의 "천체를 집어서 만지기"와 정면 충돌한다. 엔진이 순수 TS이므로 테스트가 가능하고, 훗날 규모를 키우려면 이 클래스를 통째로 워커에 옮기면 된다 — 그 문을 닫지 않는다.

### 경계 규칙 (2줄)

1. **천체의 위치·속도는 절대 React state에 들어가지 않는다.** 엔진의 `Float64Array`에만 존재하고, `useFrame`이 매 프레임 그 배열을 읽어 `InstancedMesh` 행렬에 직접 쓴다. 리렌더 0회.
2. **UI → 물리는 명령형 호출로만.** 배속 변경은 `engine.timeScale = 4`. 엔진은 React를 다시 그리게 만들지 않는다.

화면에 표시할 파생 수치(천체 수, 경과 시간, FPS)는 매 프레임이 아니라 **10Hz로 샘플링**해 state로 밀어 올린다.

### 상태 소유권

| 상태 | 소유자 |
|---|---|
| 위치, 속도, 질량, 반지름, 천체 수 | 엔진 (`Float64Array`) |
| 일시정지, 배속, 스폰 질량, 궤적 토글, 선택된 천체 id | React Context |
| 표시용 파생 수치 (천체 수·경과 시간·FPS) | React state, 10Hz 샘플링 |
| 카메라 | three.js (OrbitControls) |

## 4. 물리 엔진

### 데이터 구조 — SoA

천체당 객체를 만들지 않는다. `posX/posY/posZ/velX/velY/velZ/mass/radius/type/id`를 각각 길이 512의 `Float64Array`(일부는 정수 배열)로 **미리 할당**한다. 캐시 효율이 좋고 GC 압력이 0이다. 천체 제거는 배열 끝 원소를 빈자리로 옮기는 swap-remove로 O(1) 처리한다. 매 프레임 할당이 없다는 것이 핵심.

### 적분기 — 립프로그 (velocity Verlet)

오일러법은 궤도를 돌수록 에너지가 새어 나가 행성이 나선을 그리며 떨어지거나 튕겨 나간다. 립프로그는 심플렉틱 적분기라 에너지가 장기적으로 유계다. **안정된 궤도가 오래 안정되게 유지되는 것은 3단계의 전제 조건**이므로 여기서 아끼지 않는다.

### 중력 — 소프트닝을 넣은 O(N²)

`F = G·m₁·m₂ / (r² + ε²)^1.5 · r_vec`

두 천체가 아주 가까워지면 `1/r²`이 폭발해 천체가 광속으로 튕겨 나간다. 완화항 ε이 이 특이점을 막는다. 300개면 프레임당 약 4.5만 쌍이고, 뉴턴 3법칙의 대칭성을 이용해 절반만 계산한다.

### 시간 — 고정 스텝 누적기

화면 프레임(가변)과 물리 스텝(고정 1/120초)을 분리한다. 배속은 물리 스텝을 여러 번 도는 것으로 구현한다. 탭을 백그라운드에 뒀다 돌아오면 dt가 몇 초씩 튀므로 **dt를 0.05초로 클램프**하고 프레임당 최대 서브스텝 수도 제한한다(죽음의 나선 방지).

### 충돌 — 병합

거리 < 반지름 합이면 두 천체를 합친다.

- 질량: `m = m₁ + m₂` (보존)
- 속도: `v = (m₁v₁ + m₂v₂) / (m₁ + m₂)` (운동량 보존)
- 위치: 질량중심
- 반지름: `r = ∛(r₁³ + r₂³)` (부피 보존)
- 정체성: 더 무거운 쪽의 id·색을 유지, 가벼운 쪽은 swap-remove

밀도가 일정하다는 가정이므로 질량과 반지름은 항상 `r ∝ ∛m`으로 일관된다.

### 단위계

**G = 1인 추상 단위.** SI 단위(G=6.67e-11, 질량 1e24kg)는 부동소수점 정밀도만 낭비하고 얻는 게 없다. 사용자에게는 "소행성 / 행성 / 항성" 프리셋 이름으로 노출한다.

### 수치 안전장치

- 매 스텝 후 NaN/Infinity 검사 → 오염된 천체 제거 + 경고. 한 개의 NaN이 다음 프레임에 전 천체로 전염되는 것이 N-body의 전형적 죽음이다.
- 512개 상한 초과 시 스폰 거부, UI에 표시.

## 5. 렌더링

- **천체 — 단 하나의 InstancedMesh.** 저폴리 구체 지오메트리 1개를 512 인스턴스로. `useFrame`에서 엔진 배열을 읽어 행렬(위치+크기)과 인스턴스 색을 채운다. **draw call 1회.**
- **궤적 — 질량 상위 32개 + 선택된 천체만.** 전부 그리면 링버퍼 수백 개 갱신이 물리보다 비싸진다. 전체를 하나의 `LineSegments` 버퍼에 담아 역시 draw call 1회. 실제로 눈에 띄는 건 큰 천체의 궤도지 부스러기의 궤적이 아니다.
- **배경 — `Points`로 수천 개의 정적 별.**
- **톤 — 네온 사이파이.** 검은 배경, 발광 재질, 블룸. 블룸은 `@react-three/postprocessing`을 추가해 쓴다. 재질 트릭으로 흉내 내는 것보다 정직하게 포스트프로세싱을 쓰는 편이 결과도 낫고 코드도 짧다.

## 6. 조작

### 던지기 (새총)

1. **왼쪽 버튼**으로 빈 공간을 누른다 → 포인터 광선과 **황도면(y=0)**의 교점이 스폰 위치로 확정.
2. 누른 채 끈다 → 끈 방향의 **반대쪽**으로, 끈 길이에 비례하는 초기 속도.
3. 끄는 동안 **예상 궤적을 실선으로 미리 보여준다.** 현재 천체들의 중력장을 고정한 채 새 천체 하나만 몇백 스텝 굴려 보는 것이라 비용이 거의 없다. 이 미리보기가 "궤도에 정확히 얹는" 쾌감을 만들며, 이 단계 재미의 절반을 책임진다.
4. 놓으면 스폰.

3D 공간의 깊이 문제는 **황도면 기준 배치**로 해결한다. 실제 태양계도 거의 평면이라 자연스럽고, 생성 후엔 중력이 천체를 3D로 흔든다.

### 카메라

| 입력 | 동작 |
|---|---|
| 왼쪽 드래그 | 던지기 (카메라 아님) |
| 오른쪽 드래그 | 회전 |
| 휠 | 줌 |
| 가운데 드래그 | 팬 |

왼쪽 버튼을 온전히 던지기에 내준다. 3D 에디터의 관습과 같다.

### 선택

끌지 않고 클릭 → 선택. 정보 카드가 뜨고 카메라가 그 천체를 따라간다. **2단계(신의 손)의 모든 기능이 이 선택 위에 올라간다.**

## 7. UI 오버레이 (Canvas 위 DOM)

- 좌상단: 천체 수 / 경과 시간 / FPS
- 하단 중앙: 재생·일시정지 · 배속(0.25×~16×) · 리셋
- 우측: 스폰 질량 프리셋(소행성·행성·항성) + 미세 조정 슬라이더, 궤적 토글
- 선택 시: 천체 정보 카드

## 8. 테스트 (Vitest 신규 도입)

엔진이 순수 TS라 브라우저 없이 검증할 수 있다. 이것이 A안을 고른 진짜 이유다.

| 테스트 | 검증 대상 |
|---|---|
| 원궤도 안정성 | 2체 원궤도 100바퀴 후 반지름 1% 이내 유지 (립프로그 동작) |
| 운동량 보존 | 병합 전후 총 운동량 동일 |
| 질량 보존 | 병합 전후 총 질량 동일 |
| 부피 보존 | 반지름 합성이 `∛(r₁³+r₂³)` |
| 정면충돌 | 같은 질량·반대 속도 → 정지한 1개 |
| 결정론 | 같은 시드 → 같은 결과 (3단계 세이브/로드의 전제) |
| NaN 격리 | 오염된 천체가 다른 천체로 전염되지 않음 |

  bodies.ts         SoA 버퍼, 추가/swap-remove

## 10. 이후 단계를 위한 확장 지점

지금 구현하지는 않되, 코어가 막지 않도록 열어둘 문. 이 다섯 개 API와 하나의 플래그가 2·3·4단계 전부의 접점이다.

```ts
engine.setMass(id, m)        // 2단계: 신의 손
engine.remove(id)            // 2단계
engine.serialize(): State    // 3단계: 세이브
engine.load(state)           // 3단계: 프리셋 로드
engine.applyForce(id, vec)   // 4단계: 우주선 추력
body.type                    // 0=일반, 1=블랙홀, 2=선박
```
