# 우주 샌드박스 — 1단계 코어 설계

- 작성일: 2026-07-14
- 상태: 1단계 구현 완료 (2026-07-14)
- 범위: 4단계 로드맵 중 **1단계(코어 샌드박스)만**

> 이 문서는 살아있는 설계 문서다. 구현 중 설계가 바뀌면 코드와 함께 이 문서를 갱신한다.

## 1. 목적

천체를 던지고 그것들이 서로 끌어당기다 뭉치는 과정을 구경하는 **샌드박스 장난감**. 교육용 정확성이나 실제 태양계 재현이 목적이 아니다. 물리는 "그럴듯하고 안정적"이면 충분하며, 재미와 반응성이 우선한다.

## 2. 로드맵과 이번 범위

4개 레이어를 순차적으로 쌓는다. 각 단계는 그 자체로 완성되어 즐길 수 있고, 다음 단계는 앞 단계를 부수지 않고 얹힌다.

| 단계 | 내용 | 상태 |
|---|---|---|
| 1 | **코어 샌드박스** — N-body 중력, 던지기, 병합, 카메라, 시간 제어 | 완료 |
| 2 | 신의 손 — 선택한 천체의 질량 편집·삭제·블랙홀화 | 완료 |
| 3 | 시나리오 프리셋 — 안정된 태양계/쌍성계 로드 후 망가뜨리기 (세이브/로드) | 완료 |
| 4 | 우주선 조종 — 플레이어 선박, 추력, 스윙바이 | 이후 |

3단계 구현: 프리셋 레지스트리는 `lib/sim/scenes.ts`의 `SCENE_PRESETS`(안정된 태양계·쌍성계·블랙홀+강착원반·충돌 코스), 영속화는 `lib/saves.ts`(localStorage 슬롯 + JSON 파일 검증), UI는 `components/ui/ScenePanel.tsx`(좌상단 접이식). 엔진의 기존 `serialize()`/`load()`를 그대로 재사용한다. 설계 상세는 [`2026-07-22-scenario-presets-design.md`](2026-07-22-scenario-presets-design.md).

순서의 근거: 2·3·4는 모두 1의 물리 상태 위에서만 성립한다. 2는 코어의 상태 조작 API를 그대로 쓴다. 3의 세이브/로드는 상태 직렬화이므로 2에서 조작 API가 정리된 뒤라야 깔끔하다. 4는 카메라 모드와 입력 체계가 새로 붙는 가장 이질적인 조각이라 마지막이다.

**규모:** 동시 수백 개(상한 512). 근사 없는 진짜 N-body가 가능한 규모이며, 개별 천체를 클릭해 만질 수 있는 상한이기도 하다.

## 3. 아키텍처

```
┌─ DOM 오버레이 (React + Tailwind) ──── UI 상태만 소유
├─ Canvas 경계 (React Three Fiber) ──── 엔진을 읽어서 그리기만
└─ 시뮬레이션 엔진 (순수 TypeScript) ── 모든 물리 상태 소유
```

물리를 메인 스레드의 프레임워크 무관 클래스로 둔다. Web Worker나 GPGPU는 이 규모에서 얻는 게 없고(YAGNI), 특히 GPGPU는 2단계의 "천체를 집어서 만지기"와 정면 충돌한다. 엔진이 순수 TS이므로 테스트가 가능하고, 훗날 규모를 키우려면 이 클래스를 통째로 워커에 옮기면 된다 — 그 문을 닫지 않는다.

### 경계 규칙 (2줄)

1. **천체의 위치·속도는 절대 React state에 들어가지 않는다.** 엔진의 `Float64Array`에만 존재하고, `useFrame`이 매 프레임 그 배열을 읽어 `InstancedMesh` 행렬에 직접 쓴다. 리렌더 0회.
2. **UI → 물리는 명령형 호출로만.** 배속 변경은 `engine.timeScale = 4`. 엔진은 React를 다시 그리게 만들지 않는다.

화면에 표시할 파생 수치(천체 수, 경과 시간, FPS)는 매 프레임이 아니라 **10Hz로 샘플링**해 state로 밀어 올린다.

### 상태 소유권

| 상태 | 소유자 |
|---|---|
| 위치, 속도, 질량, 반지름, 천체 수 | 엔진 (`Float64Array`) |
| 일시정지, 배속, 스폰 질량, 궤적 토글, 선택된 천체 id | React Context |
| 표시용 파생 수치 (천체 수·경과 시간·FPS) | React state, 10Hz 샘플링 |
| 카메라 | three.js (OrbitControls) |

### 알려진 격차 / 이후 정리

이번 정리에서 고치지 않고 기록만 해 두는 두 가지:

1. **10Hz 통계 푸시가 Context identity를 바꾼다.** `setStats`가 10Hz로 새 객체를 만들어 Context에 올리므로, `useSimulation()`을 구독하는 모든 컴포넌트(Canvas 안의 씬 컴포넌트 포함)가 초당 10회씩 리렌더된다. 비용은 작고 정확성 버그도 아니지만, §3의 "리렌더 0회"는 엄밀히는 **천체 데이터**에만 해당하는 말이다. 통계를 별도 Context로 분리하면 이 문구가 문자 그대로도 성립하게 된다.
2. **`components/ui/BodyCard.tsx`가 100ms 간격으로 `engine.bodies`를 직접 읽는다.** 배열을 React state에 담지는 않으므로 "천체 위치·속도는 절대 React state에 들어가지 않는다"는 실질적인 불변식은 지켜진다. 다만 "UI는 UI 상태만 읽는다"는 문서상의 경계는 문자 그대로는 어기는 셈이다. 2단계 작업 전에 이 규칙 쪽을 수정하거나(엔진 직접 읽기를 명시적으로 허용) 카드 쪽을 고치는(파생 값을 다른 경로로 공급) 것 중 하나를 선택해야 한다.

## 4. 물리 엔진

### 데이터 구조 — SoA

천체당 객체를 만들지 않는다. `posX/posY/posZ/velX/velY/velZ/mass/radius/type/id`를 각각 길이 512의 `Float64Array`(일부는 정수 배열)로 **미리 할당**한다. 캐시 효율이 좋고 GC 압력이 0이다. 천체 제거는 배열 끝 원소를 빈자리로 옮기는 swap-remove로 O(1) 처리한다. 매 프레임 할당이 없다는 것이 핵심.

### 적분기 — 립프로그 (velocity Verlet)

오일러법은 궤도를 돌수록 에너지가 새어 나가 행성이 나선을 그리며 떨어지거나 튕겨 나간다. 립프로그는 심플렉틱 적분기라 에너지가 장기적으로 유계다. **안정된 궤도가 오래 안정되게 유지되는 것은 3단계의 전제 조건**이므로 여기서 아끼지 않는다.

### 중력 — 소프트닝을 넣은 O(N²)

`F = G·m₁·m₂ / (r² + ε²)^1.5 · r_vec`

두 천체가 아주 가까워지면 `1/r²`이 폭발해 천체가 광속으로 튕겨 나간다. 완화항 ε(`SOFTENING = 0.5`, `lib/sim/units.ts`)이 이 특이점을 막는다. 뉴턴 3법칙의 대칭성을 이용해 절반만 계산한다.

**정정(2026-07-14):** "300개면 프레임당 약 4.5만 쌍"은 틀렸다 — 이건 **서브스텝당** 쌍 수이지 프레임당이 아니다. `timeScale`이 프레임당 서브스텝 수를 그대로 곱한다(§4 "시간" 절 참고). 실측한 물리 전용(렌더링 제외) 비용은 다음과 같다:

| 천체 수 | 1× | 4× | 16× |
|---|---|---|---|
| 150 | 0.42 ms | 1.73 ms | 6.99 ms |
| 300 | 1.69 ms | 6.92 ms | 27.35 ms |
| 512 | 5.16 ms | 20.41 ms | 80.97 ms |

60fps 예산은 16.7ms다(이 표는 물리만이고 렌더링 전이다). **"512개, 60fps" 목표는 1×에서만 성립한다.** 배속을 올리면 프레임당 서브스텝이 늘어나 이 비용이 그대로 곱해지므로, 300개 이상에서 4× 이상은 예산을 넘긴다.

다만 이건 폭주가 아니라 **설계된 열화**다: `MAX_SUBSTEPS`(128)가 프레임당 작업량의 상한이고, 그 이상 밀린 시간은 버려진다(§4 "시간" 절). 그러므로 고배속에서 실제로 벌어지는 일은 "64×가 조용히 64×보다 느려진다"이지, 프레임이 죽음의 나선에 빠지는 게 아니다. 이전 버전의 이 문서는 배속이 항상 그대로 지켜진다고 암시했는데, 그렇지 않다 — 위 표처럼 물리 비용이 배속에 비례해 커지고, 상한을 넘으면 그 초과분만큼 체감 배속이 낮아진다. `MAX_SUBSTEPS`를 32에서 128로 올려(2026-07-22) 최고 배속을 16×에서 64×까지 열었다. 그 대가로 천체가 많은 씬에서 고배속의 프레임당 비용이 4배까지 커질 수 있다 — "재미(빠른 감기) 우선, 극단적 씬의 프레임 드롭은 감수"라는 장난감 성격에 따른 선택이다.

### 시간 — 고정 스텝 누적기

화면 프레임(가변)과 물리 스텝(고정 1/120초, `FIXED_DT`)을 분리한다. 배속은 물리 스텝을 여러 번 도는 것으로 구현한다. 탭을 백그라운드에 뒀다 돌아오면 dt가 몇 초씩 튀므로 **dt를 0.05초로 클램프**(`MAX_FRAME_DT`)하고 프레임당 최대 서브스텝 수도 128회로 제한한다(`MAX_SUBSTEPS`, 죽음의 나선 방지). 상한에 걸리면 밀린 시간은 버리고(누적기를 0으로 리셋), 시뮬레이션이 느려질 뿐 폭발하지 않는다. 이 상한이 하단 배속 버튼의 최고값(`64×`)이 실제로 소화되는 근거다 — `128 · (1/120) ≈ 1.07`초/프레임이 60fps에서 64×에 해당한다.

### 충돌 — 병합

거리 < 반지름 합이면 두 천체를 합친다.

- 질량: `m = m₁ + m₂` (보존)
- 속도: `v = (m₁v₁ + m₂v₂) / (m₁ + m₂)` (운동량 보존)
- 위치: 질량중심
- 반지름: `r = ∛(r₁³ + r₂³)` (부피 보존)
- 정체성: 더 무거운 쪽의 id·색을 유지, 가벼운 쪽은 swap-remove

밀도가 일정하다는 가정이므로 질량과 반지름은 `r ∝ ∛m`으로 일관된다 — **단, 다음 예외가 있다.** `radiusFromMass()`는 `MIN_RADIUS = 0.3`에서 하한 클램프를 건다(화면에서 안 보일 만큼 작아지는 것을 막기 위해서다). 아주 가벼운 천체(예: 소행성)는 이 하한에 걸려 실제 반지름이 `∛m`보다 크다. 게다가 `mergeInto()`는 병합 시 `radiusFromMass(m₁+m₂)`을 다시 계산하는 게 아니라 **이미 클램프된** `r₁`, `r₂`를 `∛(r₁³+r₂³)`으로 합성한다. 그래서 클램프가 걸린 두 천체가 병합된 뒤에는 `radius !== radiusFromMass(mass)`가 된다 — 질량과 반지름의 `r ∝ ∛m` 관계가 깨진 채로 굳어진다. 이건 버그가 아니라 "작은 천체가 안 보이게 사라지지 않는다"는 요구사항과 "병합은 그 순간의 반지름끼리 부피 보존한다"는 규칙이 함께 낳는 의도된 예외다.

### 단위계

**G = 1인 추상 단위.** SI 단위(G=6.67e-11, 질량 1e24kg)는 부동소수점 정밀도만 낭비하고 얻는 게 없다. 사용자에게는 "소행성 / 행성 / 항성" 프리셋 이름으로 노출한다.

### 수치 안전장치

- NaN/Infinity 검사는 서브스텝(`substep()`)의 **시작과 끝 두 번** 돈다. 오염된 천체는 제거하고 경고를 남긴다. 한 개의 NaN이 다음 프레임에 전 천체로 전염되는 것이 N-body의 전형적 죽음이다.
  - 서브스텝 앞: 이전 프레임 이후 버퍼가 직접 건드려졌을 수 있으므로(외부 주입, 잔여 오염 등) 이미 망가진 천체를 격리한다. `integrate()`가 내부에서 가속도를 다시 계산하므로, 서브스텝이 시작되는 시점에 이미 non-finite인 천체를 먼저 걸러내지 않으면 그 천체 하나가 **같은 호출 안에서** 다른 모든 천체의 가속도를 오염시킨다. 즉 `integrate()` 진입 전 검역만이 건강한 천체들을 살릴 수 있는 유일한 시점이다.
  - 서브스텝 끝: 그 스텝에서 새로 발생한 오염(병합, 충돌 등)을 잡는다.
  - **알려진 한계:** 이 두 번의 검사는 "스텝 경계에서 이미 오염됐거나 스텝 도중 충돌/병합으로 오염된" 경우만 잡는다. 천체가 `substep()` 진입 시점엔 정상이었다가 `integrate()` **내부의 drift 단계에서** 오버플로로 `Infinity`/`NaN`이 되는 경우는 여전히 못 막는다 — `integrate()`가 drift 직후 자체적으로 `computeAccelerations()`를 재호출하기 때문에, 그 순간 갓 오염된 천체가 같은 호출 안에서 건강한 천체들에게 오염을 퍼뜨리고, 서브스텝 끝의 `sanitize()`는 이미 전염된 그 건강한 천체들까지 함께 제거해 버린다. 겉으로는 "멀쩡했던 천체가 갑자기 사라지는" 현상으로 보인다.
    이걸 제대로 막으려면 `integrate()`를 drift 단계와 kick 단계로 쪼개 그 사이에 엔진이 검역할 지점을 넣어야 하는데, 이는 4단계 태스크에서 승인받은 `integrate()`의 표면을 재구성하는 일이라 지금은 하지 않는다 — 더구나 중력 소프트닝 덕분에 이 경로로 실제 발산이 일어나는 일은 드물다. 운영 중 `[sim] 오염된 천체 제거` 경고가 반복적으로 뜨면 이 한계를 재검토할 신호로 삼는다.
- 512개 상한(`MAX_BODIES`) 초과 시 `spawn()`이 `-1`을 반환하며 스폰을 거부한다.
  **구현 격차(일부 해소):** 반환값을 호출자(`SpawnController`)가 확인하지 않아 상한에 걸린 던지기 자체는 여전히 조용히 무시된다. 다만 `StatsHud`가 이제 천체 수를 `count / MAX_BODIES`로 표시하므로(2026-07-14), 사용자가 상한에 다가가고 있다는 것/도달했다는 것 자체는 화면에서 볼 수 있다 — "왜 방금 던진 게 안 나타나지"에 대한 직접적인 피드백(토스트 등)은 여전히 없고, 이건 2단계 작업 전에 마저 채워야 한다.

## 5. 렌더링

- **천체 — 단 하나의 InstancedMesh.** 저폴리 구체 지오메트리(`icosahedronGeometry`, detail 3) 1개를 512 인스턴스로. `useFrame`에서 엔진 배열을 읽어 행렬(위치+크기)과 인스턴스 색을 채운다. **draw call 1회.**
- **궤적 — 질량 상위 32개(`TRACKED`) + 선택된 천체만, 천체당 120개 점(`POINTS`).** 전부 그리면 링버퍼 수백 개 갱신이 물리보다 비싸진다. 전체를 하나의 `LineSegments` 버퍼에 담아 역시 draw call 1회. `engine.simTime`(엔진이 실제로 적분한 시간)이 마지막 샘플 시점보다 0.05초(`SAMPLE_DT`) 이상 전진할 때마다 샘플을 찍고, 그만큼 `lastSampleTime`을 전진시킨다 — 한 프레임에 여러 번 전진했으면(고배속) 그 배만큼 여러 번 샘플링하며, 링버퍼 길이(`POINTS`)를 넘는 샘플은 의미가 없으므로 프레임당 그 개수로 캡을 둔다. `delta * timeScale`이 아니라 `simTime`을 기준으로 삼는 이유는 `step()`이 `MAX_SUBSTEPS`에 걸려 밀린 시간을 버릴 수 있기 때문이다 — `delta * timeScale`은 엔진이 실제로 적분한 시간보다 더 많이 흘렀다고 믿을 수 있다. `reset()`으로 `simTime`이 0으로 되감기면 그 시점에 맞춰 `lastSampleTime`도 다시 동기화한다. 추적 대상은 0.5초(`RETARGET_DT`, 실시간 기준)마다 재선정한다. 실제로 눈에 띄는 건 큰 천체의 궤도지 부스러기의 궤적이 아니다.
- **배경 — drei `<Stars>`로 정적 별 4000개.**
- **톤 — 네온 사이파이.** 검은 배경, 발광 재질, 블룸. 블룸은 `@react-three/postprocessing`의 `<Bloom intensity={1.1} luminanceThreshold={0.25} luminanceSmoothing={0.3} mipmapBlur>`를 그대로 쓴다. 재질 트릭으로 흉내 내는 것보다 정직하게 포스트프로세싱을 쓰는 편이 결과도 낫고 코드도 짧다.
- **카메라 초기 위치 — `[0, 280, 495]`, fov 55.** 시작 항성계 전체(항성 + 행성 3개 + 반지름 200~230의 소행성 띠 60개)가 16:9/4:3/1:1 화면비 모두에서 15% 이상 여유를 두고 들어오도록 맞춰져 있다(`SpaceCanvas.tsx` 참고). 이 값을 줄이면 소행성 띠가 화면 밖으로 잘릴 수 있다.

## 6. 조작

### 던지기 (새총)

1. **왼쪽 버튼**으로 빈 공간을 누른다 → 포인터 광선과 **황도면(y=0)**의 교점이 스폰 위치로 확정.
2. 누른 채 끈다 → 끈 방향의 **반대쪽**으로, 끈 길이에 비례하는 초기 속도.
3. 끄는 동안 **예상 궤적을 실선으로 미리 보여준다.** 현재 천체들의 중력장을 고정한 채 새 천체 하나만 몇백 스텝 굴려 보는 것이라 비용이 거의 없다. 이 미리보기가 "궤도에 정확히 얹는" 쾌감을 만들며, 이 단계 재미의 절반을 책임진다.
4. 놓으면 스폰.

3D 공간의 깊이 문제는 **황도면 기준 배치**로 해결한다. 실제 태양계도 거의 평면이라 자연스럽고, 생성 후엔 중력이 천체를 3D로 흔든다.

### 카메라

| 입력 | 동작 |
|---|---|
| 왼쪽 드래그 | 던지기 (카메라 아님) |
| 오른쪽 드래그 | 회전 |
| 휠 | 줌 |
| 가운데 드래그 | 팬 |

왼쪽 버튼을 온전히 던지기에 내준다. 3D 에디터의 관습과 같다.

### 선택

끌지 않고 클릭 → 선택. 정보 카드가 뜨고 카메라가 그 천체를 따라간다. **2단계(신의 손)의 모든 기능이 이 선택 위에 올라간다.**

## 7. UI 오버레이 (Canvas 위 DOM)

- 좌상단: 천체 수 / 경과 시간 / FPS
- 하단 중앙: 재생·일시정지 · 배속(0.25×~16×) · 리셋
- 우측: 스폰 질량 프리셋(소행성·행성·항성) + 미세 조정 슬라이더, **무리 소환**, 궤적 토글
- 선택 시: 천체 정보 카드 (질량·반지름·속력 + **위치 고정 토글**)

### 위치 고정 (2026-07-14 추가)

선택한 천체를 제자리에 못박는다. `BodyBuffer`에 `pinned` 플래그를 두고, **적분기가 그 천체의 위치·속도 갱신만 건너뛴다.** 중력은 그대로 내뿜으므로 다른 천체들을 붙들어 두는 '닻'이 된다 — 항성을 고정해두면 뭘 던져 넣어도 안 밀린다.

- **고정이 이긴다.** 고정된 천체에 다른 천체가 부딪혀 병합되면, 합쳐진 천체는 닻 위치에 그대로 머물고 속도 0으로 계속 고정 상태다. 질량만 불어난다. 이때만 **운동량 보존을 적용하지 않는다** — 닻이 소행성 하나에 밀려나면 기능 자체가 무의미해지기 때문이다. 둘 다 고정이면 무거운 쪽의 닻 위치를 남긴다.
- 고정된 동안 **속도를 매 스텝 0으로 눌러 둔다.** 이래야 고정을 풀었을 때 그동안 쌓인 가속도로 갑자기 튀어나가지 않고 그 자리에서 자연스럽게 낙하하기 시작한다.
- 가속도는 위치와 질량으로만 계산되므로 고정 상태를 바꿔도 `accDirty`를 세울 필요가 없다.
- 화면에서는 `components/scene/PinMarkers.tsx`가 고정된 천체 주위에 호박색 고리를 그린다. 512개 전부가 고정될 수 있으므로 단일 `LineSegments`에 담아 draw call 1회로 처리한다.
- `serialize()`/`load()`가 고정 상태를 실어 나른다 (3단계 세이브/로드 대비).

### 무리 소환 (2026-07-14 추가)

한 번에 여러 천체를 뿌리는 버튼 두 개와 개수 입력 칸(1~200, 기본 50). 질량은 위에서 고른 프리셋 값을 그대로 쓰므로, 소행성으로 맞추면 부스러기 떼가, 항성으로 맞추면 항성 떼가 쏟아진다.

- **고리 뿌리기** — 가장 무거운 천체를 중심으로 반지름 120~320 밴드에 고리 모양으로 뿌린다. 속도는 그 중심에 대한 원궤도 속도(`√(GM/r)`)에 속력 ±5%·방향 ±3°의 난도를 섞는다. **완벽한 원궤도만 놓으면 서로 영원히 스치지 않아 아무 일도 일어나지 않는다** — 어긋나야 부딪히고, 부딪혀야 뭉친다. 중심 천체가 움직이고 있으면 그 속도를 물려받아 함께 따라간다. 우주가 비어 있으면 정지한 고리를 놓아 안쪽으로 붕괴시킨다(원시 행성계 붕괴).
- **혼돈 뿌리기** — 반지름 350 구 안에 위치·속도 모두 마구잡이. 대부분 빨려들거나 튕겨 나간다. 그게 목적이다.

난수 생성기는 `lib/sim/scatter.ts`가 **인자로 주입받는다.** 모듈 안에서 `Math.random`을 직접 부르면 `lib/sim` 전체가 비결정적이 되어 엔진의 결정론 보장이 깨진다. UI는 `Math.random`을, 테스트는 시드 고정 PRNG를 넘긴다.

512개 상한에 걸리면 남은 만큼만 뿌리고 멈춘다(StatsHud의 `천체 N / 512` 표시로 보인다).

## 10. 이후 단계를 위한 확장 지점

지금 구현하지는 않되, 코어가 막지 않도록 열어둘 문. 이 다섯 개 API와 하나의 플래그가 2·3·4단계 전부의 접점이다.

```ts
engine.setMass(id, m)        // 2단계: 신의 손
engine.remove(id)            // 2단계
engine.serialize(): State    // 3단계: 세이브
engine.load(state)           // 3단계: 프리셋 로드
engine.applyImpulse(id, dvx, dvy, dvz)   // 4단계: 우주선 추력 (dv = F/m·dt)
body.type                    // 0=일반, 1=블랙홀, 2=선박
```

> **2026-07-14 변경:** `applyForce`를 `applyImpulse`로 바꿨다. 힘을 받으려면 프레임 사이에 힘을 누적할 버퍼가 필요한데, 1단계에서는 아무도 쓰지 않아 죽은 코드가 된다. 추력은 `dv = F/m·dt`로 임펄스와 동등하게 표현되므로 4단계 능력에는 손실이 없다.

> **알려진 격차 (3단계 대비):** `serialize()`가 담아내는 `SerializedBody.id`를 `load()`는 무시한다 — `BodyBuffer.add()`가 항상 새 id를 발급하기 때문에 `serialize() → load()` 왕복은 id를 보존하지 않는다. 즉 3단계에서 세이브/로드 경계를 넘나드는 UI 상태(예: 선택된 천체 id)는 id가 안정적이라고 가정하면 안 된다. 3단계 작업 시 `load()`가 넘어온 id를 유지하도록 고치거나, id 대신 다른 방식(예: 순서·이름)으로 선택 상태를 이어받는 방법을 정해야 한다.
