# 블랙홀 이벤트→효과 파운데이션 설계

- 작성일: 2026-07-15
- 상태: 구현 완료 (2026-07-15)
- 범위: 블랙홀 후속 서브프로젝트 **A** — 물리 이벤트를 화면에 보이게 만드는 파운데이션 + 두 효과(증발 소멸 섬광, 블랙홀 병합 킥·잔물결)
- 선행: [2단계 블랙홀 설계](2026-07-14-black-hole-design.md) (구현 완료)

> 이 문서는 살아있는 설계 문서다. 구현 중 설계가 바뀌면 코드와 함께 이 문서를 갱신한다.

## 0. 이 서브프로젝트가 A인 이유

블랙홀 후속 아이디어는 6개였고, 서로 독립적이라 하나의 스펙에 담으면 너무 크다. 공유 부품을 기준으로 세 덩어리로 쪼갰다:

- **A (이 문서):** 이벤트→효과 파운데이션 + 증발 섬광 + 병합 킥
- **C (이후):** 가벼운 물리 디테일 팩 — 광자 구 링(1.5 r_s) + 정보 카드 시간 지연 표시
- **B (이후):** 중력 렌즈 (독립 후처리 셰이더)
- **조석 파괴(스파게티화):** 별도 대형 게임플레이 기능, 이후

A를 먼저 하는 이유: 시뮬레이션이 씬에 "사건이 일어났다"를 알리는 통로가 지금 없다. 이 통로를 놓으면 뒤의 이벤트 기반 효과(ISCO 흡수 플레어, 붕괴 섬광 등)가 전부 열린다. A는 그 아키텍처 키스톤이다.

## 1. 목적과 원칙

2단계 설계와 같은 원칙을 잇는다: **규칙을 밸런스를 위해 지어내지 않고 실제 물리에서 파생시킨다.** A가 더하는 두 사건은 물리적으로 실재한다:

- **호킹 증발의 최후:** 실제로 증발의 마지막 순간은 폭발적 복사 방출이다. 지금은 블랙홀이 바닥 질량 아래로 떨어지면 조용히 사라진다 — "물리가 치트를 정리한다"는 서사에 클라이맥스가 빠져 있다.
- **병합의 중력파 반동(킥):** 비대칭 쌍성 블랙홀이 병합하면 중력파가 운동량을 비대칭으로 실어 날라 잔여 블랙홀이 반동한다. 실제 시뮬레이션에서 잔여 블랙홀이 은하 밖으로 튕겨 나가기도 한다.

## 2. 두 조각으로 나뉜다

A는 성격이 다른 두 부분으로 이루어진다. 이 구분이 설계의 핵심이다.

- **(a) 병합 킥** — *시뮬레이션 내부의 물리 변화*다. 잔여 블랙홀의 속도를 바꾼다. 이벤트 통로가 필요 없고, `mergeInto` 안에서 결정론적으로 적용되며 Vitest로 직접 검증된다.
- **(b) 시각효과**(증발 섬광, 병합 잔물결) — 씬이 소비할 *알림*이다. 시뮬레이션 상태를 바꾸지 않는다. 여기에만 sim→scene 이벤트 통로가 필요하다.

## 3. 병합 킥 (sim 내부, 결정론적 물리)

**조건:** `collisions.ts`의 `mergeInto`에서 **블랙홀 둘이 병합할 때만**(BH + BH). 블랙홀이 일반 천체(소행성 등)를 삼킬 때는 킥을 주지 않는다 — 8000 질량 블랙홀이 질량 1 소행성을 먹었다고 덜컥이면 우스꽝스럽고, 실제로도 킥은 쌍성 블랙홀 병합에서 유의미하다.

**크기 — 피치트(Fitchett) 질량비 스케일링:**

```
q     = m_light / m_heavy          (0 < q ≤ 1)
scale = q² · (1 − q) / (1 + q)⁵
v_kick = KICK_STRENGTH · scale
```

이 법칙의 정성적 거동은 **진짜 물리**다:

- 같은 질량(q = 1) → scale = 0. 대칭이라 반동이 없다.
- 극단적 질량비(q → 0) → scale → 0. 시험입자 극한이라 반동이 없다.
- 그 사이(q ≈ 0.38 부근)에서 최대. "적당히 비대칭인 병합이 가장 세게 튄다"는 것이 눈에 보인다.

**방향 — 병합 직전 두 블랙홀의 상대속도 방향(정규화).** 이것이 실제 킥이 사는 궤도면 방향이다.

> **근사임을 명시한다.** 실제 반동의 *방향*은 블랙홀들의 **스핀 비대칭**에서 나오는데, 이 모델에는 스핀이 없다. 그래서 방향만은 상대속도로 근사한다. **크기 법칙(피치트)은 근사가 아니라 실제 물리다.** 스핀을 도입하는 별도 결정을 내리기 전까지 이 근사를 쓴다.

**운동량:** 킥은 운동량을 깬다. 그게 맞다 — 중력파가 운동량을 실어 나가고 잔여 블랙홀이 반동한다. 즉 킥은 순수 운동량 보존 병합 결과에 `v_kick`을 **더한다.**

**적용 순서:** `mergeInto`는 기존대로 운동량 보존 속도를 먼저 계산하고, 그 위에 킥 벡터를 더한다. 상대속도는 두 블랙홀의 병합 전 속도차로 계산하므로, 속도를 덮어쓰기 **전에** 상대속도를 읽어야 한다.

**엣지 — 고정(pinned):** 병합 결과가 고정이면(다른 규칙과 일관되게 고정이 이긴다) 킥에도 밀리지 않는다. 속도 0을 유지한다. 즉 킥은 pinned 처리 **전에** 더해지고, pinned 분기가 최종적으로 속도를 0으로 덮는다 — 기존 pinned 로직이 그대로 이긴다.

**상수:** `KICK_STRENGTH`를 `units.ts`에 둔다(`C`·`HAWKING_K`와 같은 자리). §7에서 조정.

## 4. 이벤트 큐 (sim→scene 통로)

**새 파일 `lib/sim/events.ts`.** 사전할당된 `EventBuffer`:

- SoA 배열: `kind`(Uint8), `x`/`y`/`z`(Float64), `payload`(Float64).
- `EventKind = { EVAPORATION: 0, MERGE: 1 }`(확장 가능한 enum — 이후 ISCO 흡수·붕괴가 여기 붙는다).
- `capacity` 상한(§7). `push(kind, x, y, z, payload)`, `clear()`, `count`, 읽기용 접근.
- **오버플로:** 한 프레임 안에서 상한을 넘으면 새 이벤트를 조용히 버린다(시각효과일 뿐이라 몇 개 누락돼도 무해하다). 버퍼는 매 스텝 비워지므로 상한 초과는 사실상 일어나지 않는다.

**엔진이 소유한다.** `engine.events: EventBuffer`. `resolveCollisions`/`applyHawking`가 이벤트를 쓸 곳이 필요하므로, 이 함수들의 시그니처에 `EventBuffer`를 넘긴다(순수 함수를 유지하면서 — 전역이나 콜백이 아니라 넘겨받은 버퍼에만 쓴다).

**방출 지점:**

- `applyHawking`: 블랙홀을 제거할 때(`next ≤ EVAPORATION_FLOOR`) → `EVAPORATION` 이벤트. 위치는 제거 직전 그 블랙홀의 좌표, payload는 소멸 직전 질량(섬광 크기).
- 블랙홀 병합(`resolveCollisions` → `mergeInto`가 BH+BH 병합을 처리한 직후) → `MERGE` 이벤트. 위치는 병합 지점, payload는 잔여 질량(잔물결 크기).

**생명주기:** `step()` 시작에서 `events.clear()`. 한 스텝의 모든 서브스텝 동안 이벤트가 쌓인다. 씬은 그 프레임에 한 번 읽는다(`Bodies`가 프레임당 `step()`을 한 번 부르므로).

**결정론:** 이벤트는 sim으로 **되먹임되지 않는다**(가속도·위치·속도를 바꾸지 않는다). 따라서 결정론에 영향이 없다. 킥은 sim 물리를 바꾸지만 그 자체가 결정론적이다(질량·속도의 순수 함수). 2단계의 결정론 테스트가 킥이 있어도 통과해야 한다.

## 5. 시각효과 (scene)

**새 컴포넌트 `components/scene/EffectsController.tsx`.** `Bodies` **뒤에** 마운트(같은 프레임의 이벤트를 본다). 엔진을 **읽기만** 한다.

- useFrame에서 `engine.events`를 읽어 **풀링된** 효과 슬롯에 스폰한다. 풀·재사용 객체는 모두 모듈 스코프 — **useFrame 안에서 할당하지 않는다.**
- **증발 섬광:** 짧게 확 퍼졌다 ~0.5초에 페이드하는 밝은 버스트. `Points`(또는 빌보드 스프라이트) 한 종류, draw call 1회. payload(소멸 질량)로 크기를 정한다.
- **병합 잔물결:** 확장하며 페이드하는 링. `InstancedMesh`(강착원반과 같은 패턴), draw call 1회. ~1초. payload(잔여 질량)로 최종 반경을 정한다.
- **발광:** `toneMapped={false}` + Bloom(`ui-conventions.md` 준수). **AdditiveBlending은 쓰지 않는다** — 강착원반이 유일한 예외로 규정돼 있고, 그 예외를 복사하지 않는다. 기본 룩이 약하면 설계-검토에서 조정한다.
- **효과 수명:** 프레임 델타로 진행한다. 순수 시각이라 프레임률 의존이 무방하다(물리가 아니다).

검은 구·강착원반과 마찬가지로 효과는 별도 상태를 갖지 않는다. 이벤트가 스폰을 촉발하고, 이후엔 스스로 나이 들어 사라진다.

## 6. 파일과 엔진 표면

**수정:**
- `lib/sim/units.ts` — `KICK_STRENGTH` 상수, 킥 크기 헬퍼(피치트 스케일링).
- `lib/sim/collisions.ts` — `mergeInto`에 BH+BH 킥, `resolveCollisions`가 `EventBuffer`를 받아 MERGE 방출.
- `lib/sim/blackhole.ts` — `applyHawking`이 `EventBuffer`를 받아 EVAPORATION 방출.
- `lib/sim/engine.ts` — `events: EventBuffer` 소유, `step()` 시작에서 `clear()`, `substep`에서 `resolveCollisions`/`applyHawking`에 버퍼 전달.
- `components/scene/SpaceCanvas.tsx` — `<EffectsController />`를 `Bodies` 뒤에 마운트.

**생성:**
- `lib/sim/events.ts` — `EventBuffer`, `EventKind`.
- `components/scene/EffectsController.tsx`.

**엔진 표면 추가:** `engine.events`(읽기용). 새 공개 메서드는 필요 없다.

## 7. 조정 가능한 숫자

- `KICK_STRENGTH` — 킥의 시각적 세기. 너무 크면 병합 잔여 블랙홀이 화면 밖으로 날아가고, 너무 작으면 안 보인다. 구현 후 사람이 브라우저에서 맞춘다.
- `EventBuffer.capacity` — 한 프레임의 이벤트 상한. 병합·증발이 한 프레임에 동시에 여럿 나는 경우는 드무므로 작게(예: 64) 잡는다.
- 효과 지속 시간(섬광 ~0.5초, 잔물결 ~1초)과 색은 시각 조정 대상.

## 8. 테스트 (Vitest)

`lib/sim`만 Vitest 대상이다(씬 효과는 사람이 브라우저에서 확인한다).

- `events.test.ts` — push/clear/count, 오버플로 시 드롭, SoA 필드 정확성.
- `collisions.test.ts` — 킥 크기 법칙(같은 질량 → 킥 0, 비대칭 → 킥 있음, 방향 = 상대속도 방향), pinned 블랙홀은 킥에도 안 밀림, BH+일반 병합은 킥 없음, BH+BH 병합 시 MERGE 이벤트 방출.
- `engine.test.ts` — 증발 시 EVAPORATION 이벤트가 올바른 위치·질량으로 방출, `step()` 시작에서 이벤트가 비워짐, **블랙홀 병합이 있어도 결정론 유지**(킥 포함).

## 9. 비목표

- 스핀·제대로 된 중력파 파형·링다운. 킥은 방향 근사 + 크기 법칙까지만.
- ISCO 흡수 플레어·자동 붕괴 섬광. 이벤트 통로는 이것들을 열어두지만, 이번 증분에서 붙이지 않는다.
- 효과의 물리적 되먹임(잔물결이 천체를 밀거나 하지 않는다). 순수 시각이다.
- 씬 효과의 자동 테스트. 브라우저 확인은 사람이 한다.

## 10. 사람의 브라우저 확인 (자동화 불가)

- 작은 블랙홀이 증발해 사라질 때 섬광이 터지는가.
- 블랙홀 둘을 병합시키면 잔여 블랙홀이 튕기고 잔물결이 퍼지는가. 같은 질량 둘은 안 튀고, 적당히 다른 질량이 가장 세게 튀는가.
- 효과가 뒤쪽 천체를 부자연스럽게 가리지 않는가. FPS가 무너지지 않는가.
- `KICK_STRENGTH`가 적절한가(잔여 블랙홀이 화면 밖으로 사라지지 않는가).
