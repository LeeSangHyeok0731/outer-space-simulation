# 조석 파괴 설계 — 파편으로 분열 (spaghettification)

- 작성일: 2026-07-15
- 상태: 승인됨 (구현 대기)
- 범위: 블랙홀 후속 서브프로젝트 **조석 파괴** — 블랙홀에 너무 가까이 간 천체가 조석력으로 여러 파편(DEBRIS)으로 찢어진다. **실제 물리를 바꾸는** 첫 후속(천체를 스폰/제거).
- 선행: [중력 렌즈](2026-07-15-gravitational-lensing-design.md) (구현 완료, B). [디테일 팩](2026-07-15-blackhole-detail-pack-design.md) (완료, C). [이벤트→효과](2026-07-15-blackhole-effects-design.md) (완료, A). [2단계 블랙홀](2026-07-14-black-hole-design.md) (완료).

> 이 문서는 살아있는 설계 문서다. 구현 중 설계가 바뀌면 코드와 함께 이 문서를 갱신한다.

## 0. 로드맵에서의 위치

블랙홀 후속 네 덩어리의 **마지막**이다: A(이벤트→효과, 완료), C(디테일 팩, 완료), B(중력 렌즈, 완료), **조석 파괴(이 문서)**. 앞의 셋은 전부 물리에 되먹임하지 않는 순수 시각효과였다. 조석 파괴는 다르다 — "찢어짐" 자체가 핵심이라, 처음으로 **실제로 천체를 부수는**(스폰/제거) 엔진 코어 변경이다. 결정론은 "난수 없음 + 고정 패턴"으로 지킨다.

## 1. 목적과 원칙

블랙홀 설계의 원칙을 잇는다: **규칙을 실제 물리에서 파생시킨다.** 조석 파괴는 블랙홀의 조석력이 천체의 자체 중력을 이길 때 일어난다. 그 경계가 **조석 반지름**이다:

```
r_t = R_body · (M_bh / m_body)^(1/3)
```

(유도: 조석 가속 `~2GM·R/d³`이 표면 자체 중력 `~Gm/R²`를 넘으면 `d < (2M/m)^(1/3)·R`. 계수는 흡수해 스케일 상수로 둔다.)

## 2. 파괴 반지름을 스케일해야 하는 이유 (C 재조정과 같은 논리)

이 우주는 광속 `C=25`로 낮춰 블랙홀을 손에 잡히는 크기로 키웠다(그래서 ISCO가 크다). 그 대가로 **실제 조석 반지름이 ISCO 안쪽에 묻힌다** — 천체가 찢어지기 전에 삼켜진다.

- 예: 최소 블랙홀(M=3000)이면 실제 `r_t ≈ 0.62·M^(1/3) ≈ 8.9`인데 `ISCO = M/104.17 ≈ 28.8`. `r_t < ISCO`라 파괴가 아예 안 일어난다.
- 게다가 이 우주는 모든 천체 밀도가 같아서(`DENSITY=1`, `R ∝ ∛m`) 질량비가 상쇄되어, `r_t`가 사실상 **블랙홀 질량에만 의존하는 껍질**이 된다(`r_t ≈ 0.62·M_bh^(1/3)`).

그래서 `C`를 재조정했듯 **조석 반지름에도 조정 계수를 곱해 ISCO 바깥의 보이는 띠로 끌어올린다**:

```
r_t = TIDAL_STRENGTH · R_body · (M_bh / m_body)^(1/3)
```

실제 공식을 유지하되(밀도가 언젠가 달라지면 천체별로 달라지도록) 계수로 스케일만 맞춘다. 파괴는 **`r_t > ISCO`일 때만** 일어난다 — 안 그러면 천체는 그냥 통째로 삼켜진다(큰 블랙홀이 별을 통째로 삼키는 실제 거동). `TIDAL_STRENGTH` 기본값은 전형적 설정(적당한 블랙홀 + 행성)에서 파괴 띠가 ISCO 바깥에 생기도록 잡고, 최종값은 사람이 브라우저에서 맞춘다.

**중요 — `R_body`는 물리 반지름을 쓴다.** `tidalRadius`는 `radiusFromMass`(표시용 하한 `MIN_RADIUS`가 걸림)가 아니라, 질량에서 직접 계산한 하한 없는 물리 반지름 `∛(3m/4πρ)`를 쓴다. 조석력은 실제 크기에 걸리지 화면에서 안 보일까 봐 부풀린 크기에 걸리지 않기 때문이다. 하한을 쓰면 극소질량 천체(질량 ≲ 0.11)의 `R_body`가 실제보다 커져 `r_t`가 비정상적으로 부풀고, 멀리서 공전하던 극소 천체가 찢겨 위의 질량 무관 껍질 성질이 깨진다. 물리 반지름을 쓰면 그 상쇄가 정확히 성립해 `r_t = TIDAL_STRENGTH · ∛(3·M_bh/4πρ)`(블랙홀 질량에만 의존)가 된다. 그래서 `tidalRadius(mBody, mBH)`는 몸체 반지름을 인자로 받지 않고 질량에서 계산한다.

## 3. 분열 모델 (`lib/sim/tidal.ts`)

천체가 파괴 조건(블랙홀의 `r_t` 안 + `r_t > ISCO`)을 만족하면, 그 자리에서 **N개 파편(`TIDAL_FRAGMENTS`, 기본 6)** 으로 대체한다(부모 제거 + 파편 스폰).

- **질량:** 균등 분할 `m/N`. **반지름:** 부피 보존 `r/∛N`.
- **위치:** 블랙홀→천체를 잇는 선(방사 방향)을 따라 가늘게 늘어놓는다 — 스파게티화(가늘게 늘어남)의 시각적 핵심. 퍼짐 폭은 부모 반지름·조정 상수 기준.
- **속도:** 부모 속도 + **결정론적** 방사 스프레드(안쪽 파편은 살짝 더 안쪽으로 빠르게 → 스트림이 늘어난다). 난수 없음.
- **질량·운동량 보존:** 파편 질량 합 = 부모 질량, 파편 운동량 가중합 = 부모 운동량(스프레드는 대칭).
- 파편은 전부 `DEBRIS` 타입(§4). 색은 부모 색 상속(어느 천체가 부서졌는지 보인다).

이후 파편은 각자 중력을 따라 움직여, 안쪽은 ISCO로 빨려들고(기존 흡수 처리), 바깥쪽은 튕겨 나가기도 한다 — 실제 조석 교란 이벤트(TDE)의 스트림 거동이 저절로 나온다.

## 4. 폭주 방지: `DEBRIS` 타입

밀도가 일정하면 파편도 부모와 **같은 조석 반지름**을 가진다(`R ∝ ∛m`이 질량비를 상쇄). 즉 파편을 그냥 두면 매 스텝 재분열해 무한 증식한다.

**해결:** `BodyType.DEBRIS = 3`을 추가하고 파편을 전부 DEBRIS로 만든다. **DEBRIS는 다시 조석 파괴되지 않는다.** 그래서 한 천체는 **최대 한 번만** 부서지고(파편 N개로 끝) 분열은 항상 유한하다.

DEBRIS는 그 외엔 일반 천체처럼 행동한다: 중력을 받고, ISCO에서 삼켜지고(기존 섬광), 서로/다른 천체와 병합도 된다(기존 `resolveCollisions`). 별도 렌더러 불필요 — `Bodies.tsx`가 반지름·색으로 그려 작은 조각으로 보인다.

## 5. 천체 예산과 서브스텝 순서

- **예산 한계:** 파편 N-1개를 새로 스폰하는데 `MAX_BODIES=512`를 넘기면 안 된다. 여유가 부족하면 **이번 파괴를 건너뛴다**(천체는 그대로 두어 ISCO에서 통째로 삼켜짐). 결정론적·안전.
- **서브스텝 순서:** `r_t`가 ISCO보다 바깥이라 천체는 공간적으로 `r_t`에 먼저 닿는다. `resolveTidalDisruption`을 **`resolveCollisions`(ISCO 흡수) 앞**에 둔다:

```
computeAccelerations → integrate → resolveTidalDisruption → resolveCollisions → sanitize
```

파괴가 일어났으면(천체 수가 바뀌었으면) 호출자는 가속도를 다시 계산한다(`accDirty`).

## 6. 결정론

- 파편 위치·속도는 부모 상태 + 고정 패턴(파편 인덱스 기반 오프셋/스프레드)으로 계산 — **난수 없음**.
- 파괴 판정은 거리 비교 — 결정론적.
- 이벤트(§7)는 물리에 되먹임되지 않는다(기존 A 원칙). 세이브/로드는 파편도 일반 천체라 이미 왕복된다(다만 파편의 id는 load 시 재발급 — 기존 한계와 동일).

## 7. 찢김 순간 연출 (`components/scene/EffectsController.tsx`)

분열 자리에 `EventKind.TIDAL = 2` 이벤트를 하나 낸다(payload = 질량). `EffectsController`가 읽어 **짧고 밝은 방사 섬광 줄기**를 그린다(스파게티화를 강조). 기존 EVAPORATION 섬광·MERGE 잔물결과 같은 풀·수명 패턴을 재사용한다. 발광은 `toneMapped=false` + Bloom(AdditiveBlending 없음). 순수 시각이라 결정론과 무관하다.

## 8. 파일과 테스트

**수정:**
- `lib/sim/units.ts` — `TIDAL_STRENGTH`, `TIDAL_FRAGMENTS`, `tidalRadius(mBody, mBH)`(물리 반지름을 질량에서 계산), `BodyType.DEBRIS = 3`.
- `lib/sim/events.ts` — `EventKind.TIDAL = 2`.
- `lib/sim/engine.ts` — `substep`에서 `resolveCollisions` 앞에 `resolveTidalDisruption` 호출, 반환 true면 `accDirty`.
- `components/scene/EffectsController.tsx` — TIDAL 이벤트 → 방사 섬광 줄기.

**생성:**
- `lib/sim/tidal.ts` — `resolveTidalDisruption(b: BodyBuffer, events?: EventBuffer): boolean`.

**테스트(`lib/sim/tidal.test.ts`, Vitest):**
- `tidalRadius` 공식 값과 `r_t > ISCO` 경계.
- 파괴 시 질량 보존(파편 합 = 부모), 운동량 보존(가중합 = 부모), 파편 개수 = N.
- DEBRIS는 다시 파괴되지 않음(폭주 방지 회귀).
- 예산 초과(count가 MAX 근처) 시 파괴 건너뜀 — 부모 유지.
- `r_t ≤ ISCO`(큰 블랙홀 등)면 파괴 안 함 — 통째 흡수 경로.
- 결정론: 같은 초기 상태 → 같은 파편 배치(반복 호출 일치).
- 씬 연출은 Vitest 대상 아님 — 사람이 브라우저로 확인.

## 9. 비목표

- **연속체/유체 스트림.** 파편은 유한 개(N)의 점질량이다. 실제 별처럼 연속으로 늘어나는 유체가 아니다.
- **밀도 다양성.** 이 우주는 밀도 일정이라 `r_t`가 블랙홀 질량 껍질로 환원된다. 천체별 밀도 차이는 도입하지 않는다(공식은 일반형을 유지해 미래 여지만 남긴다).
- **재분열(다세대 파괴).** DEBRIS는 다시 파괴되지 않는다 — 한 천체당 한 번.
- **UI 변경.** 정보 카드에 DEBRIS 라벨을 넣지 않는다(YAGNI). 파편은 일반 천체처럼 표시된다.

## 10. 조정 가능한 숫자

- `TIDAL_STRENGTH` — 파괴 띠 위치(ISCO 바깥으로 얼마나 끌어올릴지). 키우면 더 멀리서 찢어진다.
- `TIDAL_FRAGMENTS = 6` — 파편 수. 많을수록 극적이나 천체 예산을 더 쓴다.
- 파편 퍼짐 폭, 방사 속도 스프레드 — 스트림이 늘어나는 정도. 시각 조정 대상.

## 11. 사람의 브라우저 확인 (자동화 불가)

- 천체(행성 등)를 블랙홀 가까이 던지면 ISCO 바깥의 어느 거리에서 여러 파편으로 찢어지는가.
- 파편이 방사 방향으로 가늘게 늘어선(스파게티화) 스트림으로 보이는가.
- 안쪽 파편은 빨려들고(흡수 섬광), 바깥쪽은 튕겨 나가기도 하는가.
- 찢김 순간 방사 섬광 줄기 연출이 뜨는가.
- 파편이 무한 증식하지 않는가(폭주 방지 — 한 번만 부서짐).
- 큰 블랙홀은 천체를 찢지 않고 통째로 삼키는가(`r_t ≤ ISCO`).
- 파편이 많이 쌓여도(여러 번 파괴) 프레임이 유지되고 결국 흡수되어 정리되는가.
